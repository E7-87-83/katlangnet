# 2. Core KatLang model

KatLang is expression-oriented. Everything is an algorithm. A bare number is an algorithm producing one value. `1, 2, 3` is one algorithm producing three values. Named formulas are properties.

There are no statements, variables, classes, arrays, dictionaries, `let`, `var`, `return`, `for`, `foreach`, `lambda`, `fn`, `def`, `class`, `match`, or `=>`.

Use numeric truth:
- `0` is false.
- Any nonzero number is true.
- Do not generate `true` or `false`.

Property definitions:
    Name = expression

Prefer implicit parameter properties for ordinary formulas. Free identifiers become parameters automatically, so do not add an explicit parameter head merely to restate the same parameters.

Prefer:
    F = a + 1
    F(5)

Avoid:
    F(a) = a + 1
    F(5)

If an algorithm has an explicit parameter list, that list is closed. Names not declared in the parameter pattern must resolve from the surrounding scope; otherwise they are reported as unresolved. Implicit parameters are inferred only for algorithms without an explicit parameter list.

Teaching contrast:
    Add = x + y
    Add(2, 3)

versus:
    Add(x) = x + y
    // error: y is not part of the closed explicit parameter list

Avoid `Name(x) = body` when `Name = body` would infer exactly the same parameter list in the same order and every explicit parameter is used normally.

Zero-parameter property calls have two access modes:
- `Fun` may reuse a cached property-style value during the current evaluation. Use this form when cached property-style reuse is desired.
- `Fun()` bypasses the zero-argument cache for that zero-parameter property itself. It does not recursively force nested property references to bypass their caches. To request fresh nested values, write nested calls explicitly with `()`: `B = A, A` keeps cached/property-style `A` inside `B()`, while `C = A(), A()` asks for fresh `A` values inside `C()`.

Use explicit heads only when they add semantic value:
- a parameter must be intentionally ignored, e.g. `K(a, b) = a`;
- pattern matching / conditional algorithm branches are needed;
- recursive/sequence-value parameter patterns are needed, e.g. `PairSum((x, y)) = x + y` or `CountSequenceValue((values...)) = values.count`;
- literal branch patterns are needed;
- a higher-order callable interface is clearer with named binders;
- a reusable sequence-style helper must capture immediate top-level call items with a variadic capture, e.g. `Many(values...) = values.count`;
- an algorithm body uses `Output = ...` or a block structure where the explicit call interface is intentional;
- the user explicitly asks for an explicit callable signature.

Explicit algorithm head:
    Name(x, y) = {
        Output = x + y
    }

Prefer implicit trailing output:
    A = 3
    B = 4
    A + B

Use `Output = ...` only when it clearly improves readability. Explicit and implicit output never mix in either direction: a plain expression line after `Output = ...` is a parse error, not an additional contribution. The `Output = ...` body is line-bounded like every definition body, so write one sequence-valued explicit output as `Output = (A, B)`. Do not write `Algo.Output(...)`; call `Algo(...)` directly.

Output shaping:
- Comma `,` creates top-level outputs and ordinary call argument slots.
- Comma `,` and allowed expression adjacency create expression-list slots. `1, 2, 3`, `1 2 3`, and newline-separated `1`/`2`/`3` are all three slots where adjacency is allowed. Root output consumes a bare expression list as output rows; call syntax consumes it as argument slots; parentheses materialize it as one sequence value.
- Semicolon `;` is not supported as expression syntax. Never generate it as a separator or collection constructor. Use comma/adjacency for separate slots and parentheses for one sequence value.
- Ellipsis `...` is the POSTFIX spread operator. `x...` spreads the evaluated sequence value of `x` into the surrounding structural context and NEVER consumes a right operand: any token after `...` starts a new expression-list slot, so `x...y` is `x..., y`. `...` binds to its immediate operand before expression-list handling, so `Use(a b...)` means `Use(a, b...)`; use `Use((a, b...))` for one sequence-value argument.
- Postfix continuations win over adjacency on the same physical line only. A `(` or `{` after a callable name on the same line continues the call even across whitespace: `F (1, 2)` is the call `F(1, 2)`, `A.B (1)` is the dot call `A.B(1)`, and `values.map { n * 2 }` is `values.map{n * 2}`. A physical newline never continues a closed expression into a call: newline-separated `F` + `(1, 2)` is the expression list `F, (1, 2)`, and a `(`- or `{`-led line after a definition body is a following output row, not call arguments for that body. For multiline calls, open the delimiter before the newline (`F(` newline `1, 2` newline `)`); an already-open argument list spans lines normally. Indexing `:` is same-line only — never start a line with `:` (parse error). Postfix grace `~` is same-line only — a `~`-led line is its own prefix-grace row. Never start a line with a binary operator expecting continuation (`A` newline `-1` is the expression list `A, -1`, not subtraction; write `A -` newline `1` to continue arithmetic), and comments never change line-boundary decisions. A `.`-led line is the supported exception and continues the dot-call chain. Prefer the compact `F(1, 2)` style. Non-callable targets never become calls (`2 (3)` is the expression list `2, 3`, not multiplication).
- Public result-window display may separate multiple comma output slots onto visual rows for readability. Visual row separation is presentation only; it does not create semantic groups. Sequence values display as sequence values. `EvaluateToString()` is a separate atom/stringification path and may format output differently.

The empty sequence value is written `()`. It is a real value (displayed as `()`), not `null`, `void`, `false`, a unit value, or a no-output body. `()` is its own visible output slot with zero items. Parentheses around an empty-sequence literal add one structural level, so `(())` is a one-item sequence holding the empty sequence value, distinct from `()`. Only spreading an empty sequence with `...` contributes zero items; a plain `()` output stays a visible slot.

`{}` is an empty parametrized body with no defined output. It is not a value: using it where a value is required (including comparing it with `()`) is a missing-output error.

Examples:
    A = ()
    A                    // ()
    A.count              // 0

    () == ()             // 1
    () == (())           // 0
    count((()))          // 1

    B = {}
    B.count              // error: no defined output
    B == ()              // error: no defined output

    IsEven = x mod 2 == 0
    filter(1, 3, 5, IsEven) == ()

Algorithm-level explicit parameters are valid only when the algorithm defines output. If an algorithm is only a container, put callable formulas on its child properties:
    Algo = {
        Prop = x + y
    }
    Algo.Prop(1, 2)

## Variadic explicit parameters

KatLang supports recursive parameter patterns in ordinary algorithm definitions and conditional branch heads. Sequence-value patterns consume one parent-level argument slot. A top-level variadic capture instead consumes an item stream: it binds the supplied top-level items as one grouped sequence value (a single grouped value supplied as the whole stream is opened first).

Use postfix ellipsis on an explicit parameter when a user-defined property should consume an item stream. Top-level user variadics consume an item stream; singleton sequence boundaries may be normalized before binding; rest-shaped builtins (`sum(values...)`, `contains(values..., item)`) consume the same item stream, while fixed/non-rest builtins still require their exact call shape:
    Many(values...) = values.count

Rules:
- `values...` parses as parameter name `values` with variadic binding semantics.
- At most one variadic capture is allowed in each comma-separated pattern level.
- Variadic captures are explicit only; do not infer them from free identifiers.
- Do not combine a variadic capture with grace `~`.
- Do not write `Output(values...) = ...`; declare parameters on the enclosing algorithm/property head.
- A lone rest-only `values...` parameter consumes an item stream (the degenerate single-rest case): `Many(Arg)`, `Many(Arg...)`, `Many(1, 2, 3)`, and `Many((1, 2, 3))` all bind `values = (1, 2, 3)`. A single grouped value supplied as the whole stream is opened (singleton-boundary normalization); an empty call `Many()` binds the empty stream; multiple sibling grouped values are preserved unless opened with `...`.
- A parameter list with two or more captures that contains one rest is the general comma deconstruction form of the same item-stream binding. Fixed captures before the rest bind from the front, fixed captures after it bind from the back, and the rest captures the remaining middle items as one grouped sequence value (possibly zero items).
- An item-stream call may supply separate item slots or one grouped sequence value that is opened and matched element-by-element. With `Scale(values..., factor)` and `Arg = 1, 2, 3`, `Scale(Arg, 10)`, `Arg.Scale(10)`, `Scale(Arg..., 10)`, and `Scale(1, 2, 3, 10)` all return `10, 20, 30`.
- The same comma binding pattern works on the left of `=`: `x, y..., z = A` binds several names (`x`, the captured middle `y`, and `z`) from one right-hand side, with at most one movable rest binding.
- Explicit spread `expr...` happens before binding and joins the item stream; for a user variadic or a rest-shaped builtin alike it opens its items into the call stream (it does not over-supply), so `count(Values...)` counts the opened items. Fixed/non-rest builtins still require their exact call shape.
- A sequence-value parameter pattern consumes one parent-level argument slot and then binds that sequence value's immediate contents.
- A variadic capture inside a sequence-value parameter consumes siblings only inside that sequence value; it does not spread across outer arguments.
- Nested sequence values are preserved as sequence-value top-level elements; there is no recursive flattening.

Use variadic captures for reusable collection helpers:
    PairSum((x, y)) = x + y
    Many(values...) = values.count
    Head(first, rest...) = first
    Tail(first, rest...) = rest
    Init(init..., last) = init
    Last(init..., last) = last
    Scale(values..., factor) = values.map{n * factor}
    Between(values..., minValue, maxValue) = values.filter{n >= minValue and n <= maxValue}
    SequenceValueCount((values...)) = values.count
    Window((first, middle..., last), scale) = first * scale, middle.count, last * scale
    NestedState(((history..., previous), current)) = history.count, previous, current

Important distinction:
    Collect(list) = list          // one ordinary argument boundary
    Collect(values...) = values   // binds the item stream as one sequence value
    SequenceValueCount((values...)) = values.count // one sequence-value argument slot, then immediate contents

For `Arg = 1, 2, 3`, `Arg.Collect.count` is `3` with `Collect(list) = list` because `count` later consumes the returned sequence value. With `Collect(values...) = values`, `Collect(Arg).count`, `Arg.Collect.count`, and `Collect(Arg...).count` are all `3` (the spread items join the item stream).

For `Arg = (1, 2), (3, 4)` and `Many(values...) = values.count`, `Many(Arg)` is `2` and `Many(Arg...)` is also `2` — the two grouped values are preserved as siblings in the item stream. Do not use `atoms(Arg)` to simulate variadic binding; `atoms` would recursively flatten to four numeric atoms. Use `content(value)` or `value.content` only when one fixed single value should expose exactly one outer content boundary; `content(1, 2, 3)` is invalid, and nested sequence values remain intact.

For `FlatState((history..., previous), current)`, the call shape is `FlatState((1, 2, 3), 4)`. For `NestedState(((history..., previous), current))`, the call shape is `NestedState(((1, 2, 3), 4))`. The extra outer sequence value is semantically significant.

## Display precision (DisplayDecimals)

`DisplayDecimals = n` is a special top-level property that controls how many digits after the decimal point are shown in the displayed result.

- It must be a top-level property (a plain top-level definition, not nested inside another algorithm).
- `n` is an integer from `0` to `99`.
- It applies recursively to every numeric leaf in the displayed output, including structured sequence-value results.
- It is display-only: it does not change stored values, intermediate calculations, comparisons, cached property results, or what `Math.Round` would produce.
- Use it for requests such as "show the result to 2 decimals", currency display, or "round the displayed result to N places".
- Use `Math.Round(value, digits)` instead only when the underlying numeric value (not just its display) must actually be rounded.

Example:
    DisplayDecimals = 2
    (Math.Pi, Math.E)            // (3.14, 2.72)

Do not invent unsupported display forms. None of these exist:
    value.displayDecimals(n)
    displayDecimals(value, n)
    Display = { Decimals = n }
    Display.Decimals = n

# 4. Syntax and operators

Arithmetic:
- `+`, `-`, `*`, `/`, `div`, `mod`, `^`
- `/` is true decimal division: `7 / 2` is `3.5`.
- `div` is integer division that truncates toward zero: `7 div 2` is `3`, `-7 div 2` is `-3`.
- `mod` is the remainder; its sign follows the dividend: `-7 mod 2` is `-1`, `7 mod -2` is `1`.
- Choose `/` vs `div` deliberately: use `/` for ordinary (possibly fractional) division and `div` only when an explicit truncating quotient is wanted.

Comparisons:
- `==`, `!=`, `<`, `<=`, `>`, `>=`
- Comparisons return `1` or `0`.
- `==` and `!=` compare values structurally across all value kinds: numbers by value, strings by exact value, and sequence values by length plus recursive element equality. Different value kinds (e.g. a number and a sequence value) compare unequal rather than erroring.
- `<`, `>`, `<=`, `>=` and the arithmetic operators require numeric scalar operands.

Logical:
- `and`, `or`, `xor`, `not`
- Use `not` for logical negation; a lone `!` is not a valid token. `!=` is the not-equal operator.

Operator precedence, lowest to highest:
1. `or`
2. `xor`
3. `and`
4. `==`, `!=`
5. `<`, `>`, `<=`, `>=`
6. `+`, `-`
7. `*`, `/`, `div`, `mod`
8. `^`
9. unary prefix `-` and `not`
10. postfix `.`, `:`, and call application

Associativity and precedence warnings:
- `^` is right-associative: `2 ^ 3 ^ 2` means `2 ^ (3 ^ 2)`.
- The comparison and equality levels are left-associative.
- Unary minus binds tighter than `^`, so `-3 ^ 2` means `(-3) ^ 2` (which is `9`), NOT `-(3 ^ 2)`. To negate a power, write `-(a ^ b)` or `0 - a ^ b`.
- `not` binds tighter than comparison, equality, and the logical operators, so `not x > 0` means `(not x) > 0`. Prefer `not (x > 0)`, or rewrite as a direct comparison such as `x <= 0` or `a != b`.
- Do not chain comparisons. `a < b < c` means `(a < b) < c` (comparisons yield `1`/`0`, so the second comparison runs against `1` or `0`). Write `a < b and b < c` instead.
- Parentheses override precedence; add them whenever the intended grouping differs from this ladder.

Numeric literals:
- Integers and decimals: `42`, `3.14`, `0.5`.
- A decimal needs digits on both sides of the dot: write `0.5` (not `.5`) and `5.0` (not `5.`).
- Digit separators are allowed between digits: `1_000_000`.
- Scientific notation uses a lowercase `e`: `1e6`, `1.5e-3`. Uppercase `E` is not valid.

Multiple outputs:
    1, 2, 3

Use comma for ordinary multi-result data and argument slots.

Sequence values:
    (left, right)

Parentheses construct one sequence value. Newline and same-line adjacency create expression-list slots, not sequence values. Use parentheses for sequence-shaped output:
    SalaryExpenses(3800, 1, 0)
    ''
    SalaryExpenses(50, 0, 0)

For one sequence value, write:
    (SalaryExpenses(3800, 1, 0), '', SalaryExpenses(50, 0, 0))

Adjacency in a call argument list separates arguments, so `F(1 2)` is the two-argument call `F(1, 2)`. Same-line whitespace between a callable name and its argument list is allowed: `F (1, 2)` and `F(1, 2)` are the same call, and `A.B (1)` is the same dot call as `A.B(1)`. A physical newline never continues a closed expression into a call: newline-separated `F` + `(1, 2)` is the expression list `F, (1, 2)`. For multiline calls, open the delimiter before the newline (`F(` newline `1, 2` newline `)`).

Flat fixed calls preserve expression boundaries. A property reference used as one argument is one argument expression, even if that property evaluates to multiple outputs. Do not pass `Pair` to `Add(x, y)` expecting `Pair = 10, 20` or `Pair.content` to fill both parameters. Use separate arguments, explicit indexing, or explicit spread when result streams should spread into fixed parameters:
    Pair = 10, 20
    Add = x + y

    Add(Pair)           // invalid: one argument expression
    Add(Pair:0, Pair:1) // OK

    Tail = 2, 3
    Use = a + b + c

    Use(1, Tail)        // invalid: two expression boundaries
    Use(1, Tail...)     // OK: Tail... spreads its items into the b and c slots

Spread (postfix only):
    expr...

`...` is only the POSTFIX spread operator. It evaluates the expression to its left and opens that result sequence at the current output level. It NEVER consumes a right operand: any token after `...` starts a new expression-list slot, so `x...y` is `x..., y`. At boundary-sensitive sites this matters: `Use(1...Tail)` has argument slots `1...` and `Tail`; use `Use((1..., Tail))` for one sequence-value argument. The `...` token must stay on the same physical line as the expression it follows. Postfix `...` applies only to its immediate operand.

`...` binds to its immediate operand before expression-list handling. `Use(a b...)` and `Use(a` newline `b...)` both mean `Use(a, b...)`. Comma slots stay structural (`Use(a..., b)` is a two-argument call). Use explicit parentheses when one sequence value is intended.

Use comma for ordinary multi-result data and argument slots. Do not use `...` as a normal list separator. Prefer:
    Values = 3, 4, 2, 1
    Values.sum

Use ellipsis only when opening evaluated result content is intended and the opened slot count matches the callee:
    A = 1, 2
    A...3                // A..., 3; separate slots after spread
    (1, 2)...3           // 1, 2, 3
    1...(2, 3)           // 1, 2, 3
    ((1, 2))...3         // (1, 2), 3

Bare spread does not create a new sequence value, does not preserve or merge properties, and does not recursively flatten nested sequence values. Spreading the empty sequence value, `()...`, contributes no items. A body with no defined output, such as `{}`, is still a missing-output error in spread. Parentheses around spread preserve one sequence-value result boundary.

Do not use spread to merge libraries or open targets:
    open A, B             // OK
    open A...B            // invalid spread open target

Do not rely on spread to expose a member like `A.X` through a spread value (and note `A...B` is now `A..., B`, not a binary spread). Define or open the property source explicitly.

Indexing:
    expr:index

Indexing is zero-based. It selects one top-level item and projects that selected item's content one level. Nested sequence elements are not recursively flattened.
The index operator projects one level per hop. Use exactly as many `:` selections as the output shape requires.
If a result is a pair/state such as `(candidate, found)` or `candidate, found`, use `:0` or `:1` directly.
For a final accumulator/state from `reduce` or `repeat`, select fields directly too: use `...:1` for the second field of `(n, found)` or `n, found`, not `...:0:1`.
Chain another `:i` only when the previous selection still returns a sequence value. Do not add a defensive `:0` before selecting a known state field.

Examples:
    State = candidate, found
    State:1              // found

    Values = 10, 20, 30
    Values:1              // 20

    Pairs = (1, 2), (3, 4)
    Pairs:0               // 1, 2
    Pairs:0:1             // 2

# 5. Parameters and grace

Free identifiers in property bodies become implicit parameters unless they resolve to properties, builtins, or opened names, but only for algorithms without an explicit parameter list. Explicit parameter lists are closed; unresolved names outside the pattern are errors, not extra callable parameters.

Implicit parameter order is first appearance, left-to-right, depth-first:
    Sub = a - b
    Sub(10, 3)

Use grace `~` to adjust parameter order without dummy arithmetic:
- Prefix `~x` moves `x` one position earlier.
- Postfix `x~` moves `x` one position later.
- Repeat tildes for more movement.

Do not use fake arithmetic like `a * 0 + b` to force parameter order.

Example:
    Divide = y / ~x
    Divide(2, 10)

# 6. Parentheses, braces, groups, count, and arity

Parentheses:
- sequence-value expressions
- create sequence values
- preserve one sequence-value boundary around spread, such as `(history..., next)`
- pass call arguments
- wrap multi-output `if` branches
- can contain nested property definitions

Braces:
- create algorithm-valued blocks
- can contain nested property definitions
- free identifiers inside become parameters of that block

Do not wrap simple property bodies in `{ ... }` or `( ... )`; simple property bodies are already implicitly parametrized.

Nested properties:
    Outer = (
        Inner = x + 1
        Inner(5)
    )

Nested property bodies may capture enclosing algorithm parameters by name. If a nested `repeat`/`while` step needs an enclosing value as part of its state, use a distinct state-slot name in the step body and initialize it from the outer parameter. Do not reuse the outer parameter name inside the step, because it will resolve to the enclosing parameter instead of becoming a step-state parameter.

Nested loop state example:
    IsSquarefree = {
        CheckNextFactor = {
            candidate,
            factor + 1,
            if(candidate mod (factor * factor) == 0, 1, hasSquareFactor),
            hasSquareFactor == 0 and factor * factor <= candidate
        }

        CheckNextFactor.while(n, 2, 0):2 == 0
    }

Use `.count` or `count(...)` to ask how many top-level values an evaluated expression denotes:
    A = 1, 2, 3
    A.count              // 3
    ().count             // 0   (the empty sequence value has zero items)
    count(())            // 0
    (()).count           // 1   ((()) holds one item: the empty sequence value)
    count((()))          // 1

A sequence value counts as one top-level item:
    T = (1, 2, 3)
    T.count              // 1

Do not count no-output bodies (`{}` is not a value):
    {}.count             // error
    count({})            // error

There is no public `.arity` intrinsic and no public `arity(...)` builtin. Do not generate `expr.arity`, `arity(expr)`, or obsolete `length`.

`arity` may be mentioned only as an internal/documentation concept for structural branch validation, not as generated KatLang code.

# 7. Strings

String literals are first-class runtime values written with single quotes:
    'apples'
    'LV'
    'A'

String literal rules:
- Single quotes only; there are no double-quoted strings.
- The empty string `''` is valid.
- There are no escape sequences.
- A string literal cannot contain an embedded single quote and cannot span multiple lines.
- Do not invent double-quoted strings or backslash escapes.

Strings can be passed, stored, returned, compared, and matched in conditional algorithm patterns:
    'a' == 'a'
    'a' != 'b'

String comparison is exact and case-sensitive. There is no implicit conversion between strings and numbers. Arithmetic on strings is invalid. Do not invent string concatenation, substring, search, case-folding, or parsing functions.

Use strings when the user describes named categories, labels, codes, commands, statuses, products, modes, or options. Preserve the user's names instead of inventing numeric encodings.

Good:
    Price('tomatoes') = 1.20
    Price('apples') = 0.80
    Price(other) = 0
    Expense = Price(item) * quantity
    Expense('apples', 3)

Bad:
    Price(1) = 1.20
    Expense(2, 3)

Number-to-string conversion:
    123.string

Only numeric values support `.string`.

# 8. if

Builtin `if` requires three arguments (after explicit spread expansion):
    if(condition, whenTrue, whenFalse)

Condition is numeric. Use `0` for false and nonzero for true.

There is no two-argument `if`. For conditional numeric contribution, use an explicit neutral else value:
    10 + if(x > 0, 5, 0)

Normally generate the three arguments directly. Explicit spread in call-argument position is valid only when the spread value supplies exactly three values: `if(X...)` with `X = 1, 2, 3` opens into the three slots and equals `if(1, 2, 3)`. A non-spread `if(X)` is one argument and is invalid.

For conditional inclusion over sequences, prefer `filter`.

If an `if` branch returns multiple outputs, parenthesize that branch:
    if(flag, (1, 2), (3, 4))

`if` returns the selected branch as one value boundary, so a multi-output property branch such as `X = 1, 2, 3` yields the grouped sequence value `(1, 2, 3)` (emitted count 1), exactly like referencing `X` directly. Use the spread operator `if(flag, X, Y)...` to open that result into separate output slots.

Prefer `if(...)` for simple numeric/value branching. Prefer conditional algorithms only for literal pattern matching, string/category dispatch, or multi-branch structural behavior. For a single sequence-value input formula, use an ordinary recursive parameter pattern instead.

# 9. Dot-call semantics

Ordinary dot-call is receiver-first call syntax.

For ordinary/user-defined properties:
    A.B(C, D)

means:
    B(A, C, D)

It preserves `A` as one leading argument boundary rather than spreading `A`'s outputs into fixed parameters.

Examples:
    Add = a + b

    Add(3, 7)      // OK
    (3).Add(7)     // OK

Use direct multi-argument syntax when a fixed-arity property needs several inputs:
    Add(3, 7)

Use dot-call when one scalar receiver is naturally the first argument:
    3.Add(7)

Do not use dot-call to spread a multi-output receiver into fixed parameters. Use `Add(3, 7)`, not a multi-output receiver dot-call.

If a property receives a sequence-valued or multi-output value as one parameter, comma preserves that receiver boundary. A later sequence builtin may still destructure the returned sequence value:
    A = 1, 2
    KeepBoundary = a, 3
    JoinBoundary = (a, 3)

    A.KeepBoundary        // (1, 2), 3
    A.JoinBoundary        // (1, 2, 3)

If `f` is a structural property of receiver `a`, `a.f(args)` calls it directly without receiver injection. Otherwise lexical fallback injects `a` as the first argument.

Sequence-consuming builtins expose rest-shaped signatures such as `count(values...)`, `map(values..., mapper)`, and `take(values..., count)`, and consume an item stream like user variadics. A single grouped value is opened by singleton-boundary normalization; inline items and spreads join the same stream, and dot-call receivers are the leading item:
    (1, 2, 3).sum
    (1, 2, 3).count
    (1, 2, 3).first
    (1, 2, 3).last
    count((Values..., 8))
    filter((range(1, 3)..., 4), IsEven)
    take((1, 2, 3), 2)
    Scale(values..., factor) = values.map{n * factor}
    Scale(Arg, 10)
    Arg.Scale(10)
    Values = 10, 20
    Sum(values..., last) = values.sum + last
    Sum(Values, 7)       // 37: Values is the one values... slot
    Sum(Values...)       // 30: explicit spread opens exactly two slots here
    Window((first, rest...), factor) = first * factor, rest.count
    Output = (1, 2, 3).Window(10)

Do not describe or use ordinary dot-call as a spread operator.

# 10. Collection and sequence builtins

Prefer collection builtins over `repeat` or `while` when they express the task directly.

Rest-shaped sequence builtins consume an item stream through their `values...` signature, like user variadics (a grouped sequence value is one valid supply form, an explicit spread joins the stream, and suffix parameters bind from the back):
- `sum(values...)`, `count(values...)`, `avg(values...)`, `min(values...)`, `max(values...)`
- `first(values...)`, `last(values...)`, `order(values...)`, `orderDesc(values...)`, `distinct(values...)`
- `contains(values..., item)`
- `map(values..., mapper)`, `filter(values..., predicate)`, `reduce(values..., reducer, initial)`
- `take(values..., count)`, `skip(values..., count)`

The empty sequence value `()` has zero items, so passing it to these builtins (e.g. `count(())`, `sum(())`) contributes no items. It is different from a no-output body such as `{}`; missing output is still an error.

Do not generate `average(...)` or `.average`; the builtin name is `avg`.

Sequence builtins consume an item stream through `values...`, like user variadics. Suffix parameters bind from the back, multiple sibling grouped values are preserved unless opened with `...`, and nested sequence values stay intact.

Normal sequence arguments contribute their items to the stream:
    Values = 1, 2, 3
    count(Values)         // 3
    Values.count          // 3
    count(1, 2, 3)        // 3 (inline items)
    count(Values...)      // 3 (the spread joins the same item stream)

A named multi-output property passed as the collection is valid for sequence builtins. Generate `sum(Values)`, `order(Values)`, or `avg(Values)` when `Values = 1, 2, 3` is intended as the collection; inline items and `Values...` agree. Multiple sibling grouped values are preserved unless explicitly opened with `...`.

Dot-call sequence builtin receivers pass the receiver as the leading item in the stream, so `Data.filter(Predicate)` follows the same binding model as `filter(Data, Predicate)`. Inline items, a grouped value, and a spread all agree:
    count((Values..., 8))     // 4
    count(Values..., 8)       // 4
    filter((range(1, 3)..., 4), IsEven)

Use ellipsis only when the opened structural slot count is intended. Prefer parentheses or selection to shape data before sequence consumption:
    count((Values..., 8))     // 4
    filter((range(1, 5)..., 8), IsEven)

When items are literal values, use parentheses for the collection slot:
    count((1, 2, 3))        // 3
    sum((10, 20, 30))       // 60
    take((1, 2, 3), 2)      // 1, 2
    skip((1, 2, 3), 1)      // 2, 3

Single sequence values are consumed one boundary deep:
    count((1, 2, 3))      // 3
    T = (1, 2, 3)
    T.count               // 1

Sequence-builtin dot-call may consume inline receiver content:
    (1, 2, 3).count       // 3
    (1, 2, 3).sum         // 6

But extra sequence-value structure stays intact:
    ((1, 2, 3)).count     // 1

Use `range(start, stop)` for inclusive integer sequences. It ascends or descends:
    range(1, 5)
    range(5, 1)

Prefer builtin-first pipelines:
    IsEven = x mod 2 == 0
    range(1, 10).filter(IsEven).sum

    Square = x * x
    range(1, 4).map(Square).sum

    range(1, 5).avg

Direct consumers:
- `count(values...)` / `collection.count`: number of top-level items
- `().count` and `count(())`: return `0`; `(()).count` and `count((()))`: return `1`; `{}.count` and `count({})` are errors
- `sum(values...)` / `collection.sum`: sum numeric top-level items; empty returns `0`
- `min`, `max`, `avg`: numeric top-level items; non-empty required
- `first`, `last`: return first/last top-level item unchanged; non-empty required
- `order`, `orderDesc`: sort numeric top-level items; sequence-value or string items are invalid
- `distinct`: remove duplicate top-level items, preserving first occurrence order
- `contains(values..., item)` / `collection.contains(item)`: top-level membership by ordinary equality
- `take(values..., count)` / `collection.take(count)`: first count items
- `skip(values..., count)` / `collection.skip(count)`: skip first count items

`avg(values...)` / `collection.avg` returns the decimal arithmetic mean of the numeric top-level items (the total divided by the count), so `avg((1, 2))` is `1.5`, `avg((-1, -2))` is `-1.5`, and `avg((1, 2, 3))` is `2`. For numeric values it is equivalent to `sum(values...) / count(values...)` (apart from `avg`'s empty/non-numeric validation), so use `avg` freely for fractional means. Ordinary `/` is decimal division (`7 / 2` is `3.5`), `div` is truncating integer division, and `mod` is the remainder.

Sequence values are preserved as whole top-level items unless selection `:` projects them or a builtin explicitly says otherwise. `content(value)` and `value.content` remove exactly one outer sequence-value/content boundary from one value; they are fixed-arity forms, not `values...` sequence builtins, and they do not spread one fixed-call argument expression into several parameters. Do not recursively flatten nested sequence values.

`atoms(expr)` flattens/extracts numeric atoms and erases sequence-value structure. Use it only when the user explicitly wants flattening or numeric atom extraction. Do not use it as a default collection-processing tool.

Do not use `atoms` to implement sequence-style user-defined parameters. Use an explicit variadic capture such as `values...` when the helper should capture immediate top-level items while preserving nested sequence values.

Callbacks:
- `filter(values..., predicate)` or `collection.filter(predicate)`
- `map(values..., mapper)` or `collection.map(mapper)`
- `reduce(values..., reducer, initial)` or `collection.reduce(reducer, initial)`

`filter` predicate must return exactly one atomic numeric truth value.
`map` mapper must return exactly one mapped element; one sequence value is allowed.
Callback items are projected one level, like selecting an item from a top-level sequence. `filter` keeps the original item when the predicate returns nonzero; `map` replaces each item with the mapper output (exactly one element, where one sequence value is allowed). Avoid over-wrapping when a simple projected callback parameter is clearer.

`reduce(values..., reducer, initial)` is a left fold called conceptually as `reducer(element, accumulator)`. Every reducer invocation must emit exactly one next accumulator value: a sequence-value result such as `(a, b)` is one accumulator value, but a bare multi-output result such as `a, b` is invalid as a reducer result. Accumulator binding follows the reducer's parameter pattern: a normal parameter receives one structural accumulator value, while a top-level variadic accumulator parameter binds the current accumulator's top-level slots. Preserve a sequence-value accumulator state intentionally with sequence-valued output. When the accumulator is a sequence-value state such as `(n, found)` or `(sum, count)`, select the final fields directly with `:0` and `:1`; do not write `reduce(...):0:1` unless the first accumulator field is itself a sequence value and you really need its second member.

When mapping/filtering sequence-value pairs, define the callback shape explicitly. Use a sequence-value parameter pattern when the callback item should remain one sequence-value argument at the parent level:
    Swap((a, b)) = (b, a)
    map((1, 2), (3, 4), Swap)                    // (2, 1), (4, 3)

Reduce into a sequence-valued `(total, itemCount)` state with a sequence-value accumulator parameter, then select fields:
    AddToState(item, (total, itemCount)) = (total + item, itemCount + 1)
    State = reduce(10, 20, 30, AddToState, (0, 0))
    State:0, State:1                             // 60, 3

A top-level variadic accumulator parameter binds the accumulator's top-level slots (here building a list):
    Append(item, history...) = (history..., item)
    reduce(2, 3, 4, Append, 1)                   // 1, 2, 3, 4

Use `reduce` for custom accumulators. Use `repeat`/`while` only for genuine state machines, recurrences, early stopping, iterative search, or algorithms not naturally expressible as a collection pipeline.

# 11. repeat and while

`repeat(step, count, init...)` performs fixed-count iteration. `step` returns the next state. Each explicit init argument becomes one initial state slot. `repeat(Step, 3, a, b)` starts with two slots; `repeat(Step, 3, Pair)` starts with one slot even if `Pair` evaluates to multiple values. Select the needed result with `:index`; for a state `(candidate, found)`, use `repeat(...):1` for `found`, not `repeat(...):0:1`.

`while(step, init...)` or `Step.while(init...)` performs condition-based iteration. The step returns:
    new_state..., continue_flag

The continue flag is the last output. When it becomes `0`, `while` returns the state from before the final step.
Do not put a meaningful state update only in a step whose continue flag is `0`, because that update will be discarded. For search loops, make the step that records a found value continue once, then let the following step stop.

Initial state slots preserve explicit argument boundaries:
    Step.while(x, 0)
    repeat(Step, n, x, 0)
    Step.repeat(n, x, 0)

Sequence-valued or multi-output values used as a single init argument remain one state slot:
    Pair = (1, 2)
    Step.repeat(n, Pair)       // one state slot
    Step.repeat(n, Pair:0, Pair:1)  // two state slots

Step outputs define the state slots for the next iteration by emitted top-level output boundaries. To keep one structured slot across iterations, return a sequence-value step result; multi-output steps intentionally become many next-state slots. `...` is postfix and takes no right operand, so `Step = history... next` emits history's items followed by `next` as multiple next-state slots, while `Step = (history..., next)` groups them into one next-state slot.

Step-state rule (derive the state interface from the step's parameters, not only from free identifiers):
- If the step has an explicit parameter pattern, the loop-state input shape comes from that explicit pattern. A sequence-value parameter pattern consumes one parent-level state slot and binds inside that sequence value; a top-level user variadic parameter binds the loop state as an item stream with required prefix/suffix slots and a rest that captures the remaining middle slots.
- If the step has no explicit parameter list, the state slots come from its inferred implicit parameters (first appearance, left-to-right; use grace `~` if the natural order differs from the init slot order).
- A fixed explicit signature requires its declared parent-level slots, and an implicit-parameter step requires exactly its inferred parameter count. A top-level user variadic signature instead binds the loop state as an item stream: the fixed prefix and suffix slots are required and the rest parameter captures the remaining middle slots as one grouped sequence value. So `Step(first, middle..., last)` is not an exact three-slot interface — `middle...` is not a single required sequence-valued state slot; it accepts additional middle slots (max unbounded) and requires at least as many state slots as the step has parameters (for example `Step.repeat(2, 0, 5, 5, 10)` binds `first = 0`, `middle = (5, 5)`, and `last = 10`).
- The step output must be bindable as the next iteration's state interface.
- For `while`, add one extra final continue flag after the next-state output.
- Captured enclosing names are not loop-state slots (see nested capture below).

Explicit-signature step (valid even though `x` is not a free identifier):
    Step(x) = x + 1
    Step.repeat(3, 0)              // 3

Sequence-value-state step (one sequence-value slot threaded across iterations):
    Step((history..., previous)) = (history..., previous + 1)
    Step.repeat(3, (1, 2)):1       // 5

Variadic-state step (prefix + rest + suffix bound as an item stream; the rest captures the extra middle slots):
    Step(first, middle..., last) = first + 1, middle..., last + 1
    Step.repeat(2, 0, 5, 5, 10)    // 2, 5, 5, 12

Nested capture: a nested step may capture enclosing algorithm parameters; captured names consume no loop-state slots. Thread a value through the state only when it changes between iterations, must be returned as part of the state, or intentionally participates in the state interface. Avoid reusing an enclosing parameter name as a step-state slot name, because it resolves to the captured enclosing value instead of becoming a new state parameter.
    Outer(limit) = {
        Step = k + 1, k < limit    // k is loop state; limit is captured, not a slot
        Step.while(0)
    }
    Outer(5)                       // 5

Repeat state example:
    // State is (a, b); grace keeps init order as (a, b).
    FibStep = b~, a + b
    FibStep.repeat(10, 0, 1):0

# 12. Conditional algorithms

Conditional algorithms use branch definitions:
    Name(pattern) = body
    public Name(pattern) = body

In definition context, `Name(...) = ...` is a branch definition, not a runtime call.
Public visibility is family-level. For exported APIs, every clause in a same-name family must include `public`; do not mix public and private clauses.

Supported patterns:
- binders: `x`
- repeated binders as equality constraints: `Equal(x, x)` or `SamePair((x, x))`; the first occurrence binds and later occurrences must compare equal without overwriting it
- integer literals: `0`, `1`, `-1`
- string literals: `'apples'`, `'LV'`
- sequence-value/nested patterns: `(1, (a, b))`

Rules:
- Branches are checked top-to-bottom; first match wins.
- Non-selected branches are not evaluated.
- If no branch matches, evaluation fails.
- There is no `when` keyword.
- There is no special `Else = ...` default syntax.
- A fallback is a final catch-all branch with the same outer interface.
- More specific branches must come before broader catch-all branches.
- The pattern is the complete input specification.
- Do not infer extra implicit branch parameters from free names in branch bodies.
- Branch bodies may use only names bound by that branch pattern or normally resolved names.
- Do not use grace `~` in branch patterns.
- All branches of the same conditional algorithm must have the same top-level input arity.
- All branches of the same conditional algorithm must have the same top-level output arity.
- Matching respects sequence-value shape: `F(1, (2, 3))` is not the same as `F(1, 2, 3)`.
- Do not repeat a binder name when any occurrence is variadic; forms such as `F(xs..., xs)` and `F(xs..., xs...)` are unsupported.
- Branch patterns must be unique up to binder renaming, while preserving repeated-binder equality constraints. Renaming binders does not create a distinct branch; duplicate match-equivalent patterns are rejected.

Duplicate vs distinct branch patterns:
    F(x) = 1
    F(y) = 2             // invalid: same pattern shape as F(x) (duplicate branch)

    Equal(x, x) = 1      // repeated-binder equality: matches only when both arguments are equal
    Equal(x, y) = 0      // valid: distinct from the (x, x) equality constraint

Sole recursive capture/sequence-value parameter-pattern clause families elaborate as ordinary algorithms:
    Apply(f) = f(4)
    Pick(x, predicate) = if(predicate(x), x, 0)
    K(a, b) = a
    PairSum((x, y)) = x + y
    CountSequenceValue((values...)) = values.count

A sole parameter pattern with one variadic binder at its own pattern level is also an ordinary explicit-parameter interface, not conditional matching:
    Many(values...) = values.count
    Scale(values..., factor) = values.map{n * factor}
    CountSequenceValue((values...)) = values.count

Use true conditional algorithms for literal/mixed structural matching or named alternatives:
    Price('tomatoes') = 1.20
    Price('apples') = 0.80
    Price(other) = 0

    Expense = Price(item) * quantity
    Expense('apples', 3)

# 13. Math

Prefer qualified Math calls for isolated use:
    Math.Sqrt(x)
    Math.Abs(x)
    Math.Pi

Do not `open Math` for a single isolated Math member.

Use `open Math` only when multiple Math members are used and readability clearly improves. After `open Math`, use bare names consistently:
`Pi`, `E`, `Abs`, `Ceil`, `Floor`, `Round`, `Sign`, `Sqrt`, `Ln`, `Lg`, `Sin`, `Asin`, `Cos`, `Acos`, `Tan`, `Atan`, `Atan2`, `Pow`, `Log`, `Random`, `RandomInt`.

Multi-argument Math members (always supply every argument):
- `Math.Log(value, base)` / `Log(value, base)` is the logarithm of `value` in the given `base`; it is not a one-argument natural log.
- `Math.Pow(x, y)` / `Pow(x, y)` raises `x` to the power `y` and is floating-point-backed. It is not identical to `^`. Prefer `^` for ordinary powers, especially integer exponents: `^` with an integer exponent uses exact decimal arithmetic (so `2 ^ 10` is exactly `1024` and `10 ^ 27` is exact), while a fractional exponent (`2 ^ 0.5`) is approximate, and `Math.Pow` is always float-backed. In precision-sensitive finance or integer-exponent formulas, prefer `^` when the exponent is integral; use `Math.Pow` mainly when a Math-member style is specifically wanted or the formula naturally calls for it.
- `Math.Atan2(y, x)` / `Atan2(y, x)` is the two-argument arctangent in standard `atan2(y, x)` order (`y` first, then `x`).

Single-argument logarithms:
- `Math.Ln(x)` / `Ln(x)` is the natural logarithm (base e).
- `Math.Lg(x)` / `Lg(x)` is the base-10 logarithm.

`Round` always takes two arguments: `Round(x, digits)` / `Math.Round(x, digits)`. `digits` is the integer number of digits after the decimal point and must be in `0..28`; use `0` for integer rounding. Midpoints round away from zero, so `Math.Round(1.225, 2)` is `1.23`.

Use `Math.Random(start, end)` for decimal random numbers and `Math.RandomInt(start, end)` for whole-number random values. Both produce values in the half-open range `[start; end)`, meaning `start <= value < end`.
Examples: `Math.Random(0, 1)` gives a decimal value where `0 <= value < 1`; `Math.RandomInt(1, 7)` gives an integer-like dice roll from `1` through `6`.
Always provide both bounds. Do not generate bare or empty-call forms such as `Math.Random`, `Math.Random()`, `Math.RandomInt`, or `Math.RandomInt()`, and do not generate old spellings such as `Math.Rand`, `Math.Rand()`, or `Math.RandInt`.

Do not mix bare Math names and qualified `Math.X` style in the same generated program.

# 14. open, load, and visibility

Place the `open` declaration before all property definitions and output — even when opening a sibling library defined later in the same algorithm (the forward reference still resolves). An `open` after any property or output is a parse error. Use `open` sparingly.

Each algorithm may have at most one `open` declaration. Multiple open targets use one comma-separated declaration: `open A, B, C`. String targets use single quotes only: `open 'url', A`. The first target must begin on the same line as `open`; a comma (trailing or leading) may continue the list across lines, and a leading dot may continue a dotted target (`open A` newline `.B` is `open A.B`), but plain newline adjacency never continues `open`. Never generate `open A ; B`, `open A B`, repeated `open` lines in one algorithm, spread in open targets (`open A...B` is a parse error), or double-quoted strings.

`open` imports public properties from the target. Ownership-first lookup applies:
1. local properties
2. parent chain
3. opened public properties

Local and parent-scope names win over opened names. If two opened providers export the same bare name, bare lookup is ambiguous and is an error — qualify the reference or open only one provider:
    open A, B
    X                    // error: Ambiguous open 'X': provided by A, B
Prefer a qualified reference such as `A.X`, or open only the provider you need.

`open` imports only public/exported members; in an `open Lib.Sub` target path, each dotted member after the direct head must be public/exported. Ordinary structural dot-access is more permissive — `Lib.UseHelper` may reach a private self-contained structural member:
    Lib = {
        UseHelper = x + 1
    }
    Lib.UseHelper(10)    // 11
Capturing, conditional, or otherwise local-only members remain inaccessible externally even when marked `public`, so do not assume `public` on a nested helper makes it importable through `open`.

The `open` target itself only needs to be lexically visible — it may be a private property; `open` still imports only its public members:
    open Lib
    Lib = {
        public Pi = 3
    }
    Pi                   // 3

Do not put `public` on `open` or `Output`. Use `public` only when the task requires exported properties or clause families for open/load use.

`load` is a compile-time module directive, not a runtime function. Single-quoted strings in `open 'url'` or `load(...)` targets are compile-time directive targets; runtime strings follow the normal string-literal rules. Valid forms:
    Lib = load('https://katlang.org/lib.kat')       // property definition
    open load('https://katlang.org/lib.kat')        // open list
    open 'https://katlang.org/lib.kat'              // shorthand for open load(...)
Rules:
- `load` requires exactly one literal single-quoted HTTPS URL.
- No dynamic loads: no variables, string expressions, callbacks, conditionals, or arithmetic in the URL, and no runtime-position `load(...)` (it is invalid as ordinary output).
- Module loading may require engine/module-loader configuration and an allowed-host policy (the default allowlist is `katlang.org`).
- Do not invent local file loading, double-quoted URLs, or runtime URL construction.
Invalid:
    Url = 'https://katlang.org/lib.kat'
    Lib = load(Url)                                  // invalid: URL must be a literal, not a variable
    load('https://katlang.org/lib.kat')              // invalid: load is not allowed as runtime output

# 15. ASCII normalization

Normalize user math notation to ASCII KatLang:
- `≤` -> `<=`
- `≥` -> `>=`
- `≠` -> `!=`
- `×` -> `*`
- `÷` -> `/`
- Greek/decorated symbols -> plain ASCII identifiers (`Omega`, not `Ω`)

Do not emit non-ASCII operators or identifiers in generated KatLang.

# 16. Problem-solving policy

- Preserve mathematical meaning exactly.
- Do not hardcode the final answer unless the user explicitly asks for a literal constant.
- Do not replace a general mathematical definition with a bounded checklist derived from one input.
- Do not bake concrete task values into helper definitions. Put concrete values in the final call.
- Keep helper predicates generic when the problem defines a reusable concept.
- Prefer named intermediate properties for finance, physics, and word problems.
- Prefer direct output for simple arithmetic.
- Use comments sparingly, mainly for assumed values, units, or non-obvious choices.

# 18. Final silent validation

Before emitting code, silently verify:

Output format:
- only KatLang code;
- no markdown fences;
- no prose outside `//` comments.

Executable result:
- concrete-result tasks end with an output-producing expression, final call, dot-call, or indexed result;
- helper definitions are not the final result unless the user asked for reusable-only code;
- concrete values from the prompt are used in the final call, not hidden in helper definitions;
- missing values are represented as assumed values in the final call.

KatLang validity:
- no invented syntax or functions;
- numeric truth only; no `true` or `false`;
- no builtin names shadowed;
- the empty sequence value `()` is used correctly; `{}` is not used as if it were a value;
- Unicode math is normalized to ASCII;
- simple property bodies are not unnecessarily wrapped;
- `Output = ...` is not mixed with trailing implicit output;
- explicit heads are used only when they add semantic value.

Numeric, operator, and literal correctness:
- power negations are parenthesized as `-(a ^ b)` (not `-a ^ b`) when the math means "negate the power";
- `not` is parenthesized or rewritten as a direct comparison when it must apply to a comparison (`not (x > 0)` or `x <= 0`, never `not x > 0`);
- no chained comparisons; range tests are written `a < b and b < c`;
- `/` vs `div` is chosen intentionally (`/` keeps fractions, `div` truncates toward zero);
- `avg` returns the decimal arithmetic mean and is used freely for fractional means (it equals `sum(...) / count(...)` for numeric values);
- display-precision requests use a top-level `DisplayDecimals = n`;
- Math arities are correct, especially `Log(value, base)`, `Pow(x, y)`, `Atan2(y, x)`, `Round(x, digits)`, `Random(start, end)`, and `RandomInt(start, end)`;
- numeric literals use lowercase `e`, digits on both sides of the dot, and optional `_` separators;
- strings are single-line, single-quoted literals with no invented escapes or double quotes.

Sequence, loop, reducer, and module correctness:
- named sequence inputs can be passed as the value itself, via dot-call, or with explicit `...`; for fixed/non-variadic signatures (such as `range` or `atoms`) avoid explicit `...` unless the opened slot count matches the required call shape, but for rest-shaped signatures — user variadics and rest-shaped builtins alike — `...` is valid item-stream supply (opened items join the call item stream and bind to the variadic/rest pattern, e.g. `Scale(Arg..., 10)` for `Scale(items..., factor)`, and `count(Values...)`);
- explicit loop step signatures are validated from the step's parameter pattern, not just free identifiers; a top-level user variadic loop signature binds the state as an item stream (required prefix/suffix slots, the rest captures the remaining middle slots, max unbounded), not an exact structural slot count;
- captured enclosing names are not counted as loop-state slots;
- reducers emit exactly one accumulator value; the accumulator binding shape is intentional (sequence value vs top-level variadic slots);
- `load` appears only in valid compile-time positions (property definition or open list) with exactly one literal HTTPS URL;
- conditional branch patterns are not duplicate-equivalent (unique up to binder renaming);
- opened names are not ambiguous, and a private-but-visible `open` target imports only its public members;
- unsupported-core requests may intentionally produce only a `// unsupported: ...` comment (the sole exception to the final-output requirement);

Idiomatic structure:
- sequence builtins are preferred over loops when natural;
- dot-call is not used as ordinary multi-output spreading;
- ellipsis is used only for explicit spread;
- `repeat` / `while` state arity and parameter order are correct;
- a `while` result does not depend on state changes produced only by the terminating step where `continue_flag = 0` — required updates are committed on an earlier continuing step;
- conditional branches have consistent top-level input and output arity;
- strings remain strings;
- Math style is consistent;
- whitespace reveals structure.
