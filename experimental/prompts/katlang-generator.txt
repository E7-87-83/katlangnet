# 2. Core KatLang model

KatLang is expression-oriented. Everything is an algorithm. A bare number is an algorithm producing one value. `1, 2, 3` is one algorithm producing three values. Named formulas are properties.

There are no statements, variables, classes, arrays, dictionaries, `let`, `var`, `return`, `for`, `foreach`, `lambda`, `fn`, `def`, `class`, `match`, or `=>`.

Use numeric truth:
- `0` is false.
- Any nonzero number is true.
- Do not generate `true` or `false`.

Property definitions:
    Name = expression

Prefer implicit parameter properties for ordinary formulas. Free identifiers become parameters automatically, so do not add an explicit parameter head merely to restate the same parameters.

Prefer:
    F = a + 1
    F(5)

Avoid:
    F(a) = a + 1
    F(5)

If an algorithm has an explicit parameter list, that list is closed. Names not declared in the parameter pattern must resolve from the surrounding scope; otherwise they are reported as unresolved. Implicit parameters are inferred only for algorithms without an explicit parameter list.

Teaching contrast:
    Add = x + y
    Add(2, 3)

versus:
    Add(x) = x + y
    // error: y is not part of the closed explicit parameter list

Avoid `Name(x) = body` when `Name = body` would infer exactly the same parameter list in the same order and every explicit parameter is used normally.

Use explicit heads only when they add semantic value:
- a parameter must be intentionally ignored, e.g. `K(a, b) = a`;
- pattern matching / conditional algorithm branches are needed;
- recursive/grouped parameter patterns are needed, e.g. `PairSum((x, y)) = x + y` or `CountGroup((values...)) = values.count`;
- literal branch patterns are needed;
- a higher-order callable interface is clearer with named binders;
- a reusable sequence-style helper must capture immediate top-level call items with a variadic capture, e.g. `Many(values...) = values.count`;
- an algorithm body uses `Output = ...` or a block structure where the explicit call interface is intentional;
- the user explicitly asks for an explicit callable signature.

Explicit algorithm head:
    Name(x, y) = {
        Output = x + y
    }

Prefer implicit trailing output:
    A = 3
    B = 4
    A + B

Use `Output = ...` only when it clearly improves readability. Do not mix `Output = ...` with trailing implicit output in the same algorithm. Do not write `Algo.Output(...)`; call `Algo(...)` directly.

Use `empty` as the canonical explicit empty output. It is a reserved builtin constant that emits zero top-level values; do not define, shadow, or call it, and never generate `empty()`. `empty` is not `null`, `void`, `false`, a unit value, or an empty body.

Do not use `()` or `{}` for empty output. `()` is an empty non-parametrized body with no defined output, and `{}` is an empty parametrized body with no defined output. Both are missing-output errors when evaluated as values. Use `(empty)` or `{empty}` only when an algorithm/body form should explicitly return empty output.

Examples:
    A = empty
    A.count              // 0

    A = ()
    A.count              // error: no defined output

    B = {}
    B.count              // error: no defined output

    IsEven = x mod 2 == 0
    filter(1, 3, 5, IsEven) == empty

Algorithm-level explicit parameters are valid only when the algorithm defines output. If an algorithm is only a container, put callable formulas on its child properties:
    Algo = {
        Prop = x + y
    }
    Algo.Prop(1, 2)

## Variadic explicit parameters

KatLang supports recursive parameter patterns in ordinary algorithm definitions and conditional branch heads. Grouped patterns consume one parent-level argument slot. Variadic captures consume sibling values at their own pattern-list level.

Use postfix ellipsis on an explicit parameter when a user-defined property should bind sibling values from the call item stream:
    Many(values...) = values.count

Rules:
- `values...` parses as parameter name `values` with variadic binding semantics.
- At most one variadic capture is allowed in each comma-separated pattern level.
- Variadic captures are explicit only; do not infer them from free identifiers.
- Do not combine a variadic capture with grace `~`.
- Do not write `Output(values...) = ...`; declare parameters on the enclosing algorithm/property head.
- Normal captures before the variadic capture at the same pattern level bind from the front of that level's item stream.
- Normal captures after the variadic capture at the same pattern level bind from the back.
- The variadic capture captures all remaining middle items at that level.
- In flat variadic user calls, ordinary argument segments are assigned to prefix, variadic, and suffix slots first; segments assigned to the variadic capture then supply their emitted top-level items.
- Explicit sequence supply `expr...` happens before binding and can satisfy prefix or suffix slots. Use it when supplied items should participate in prefix/suffix binding.
- A grouped parameter pattern consumes one parent-level argument slot and then binds that group's immediate contents.
- A variadic capture inside a grouped parameter consumes siblings only inside that group; it does not spread across outer arguments.
- Nested grouped values are preserved as grouped top-level items; there is no recursive flattening.

Use variadic captures for reusable collection helpers:
    PairSum((x, y)) = x + y
    Many(values...) = values.count
    Head(first, rest...) = first
    Tail(first, rest...) = rest
    Init(init..., last) = init
    Last(init..., last) = last
    Scale(values..., factor) = values.map{n * factor}
    Between(values..., minValue, maxValue) = values.filter{n >= minValue and n <= maxValue}
    GroupedCount((values...)) = values.count
    Window((first, middle..., last), scale) = first * scale, middle.count, last * scale
    NestedState(((history..., previous), current)) = history.count, previous, current

Important distinction:
    Group(list) = list          // one ordinary argument boundary
    Group(values...) = values   // immediate top-level item capture
    GroupedCount((values...)) = values.count // one grouped argument slot, then immediate contents

For `Arg = 1, 2, 3`, `Arg.Group.count` is `1` with `Group(list) = list`. With `Group(values...) = values`, both `Group(Arg).count` and `Arg.Group.count` are `3` because the `Arg` segment is assigned to the flat variadic capture.

For `Arg = (1, 2), (3, 4)` and `Many(values...) = values.count`, `Many(Arg)` and `Many(Arg...)` are both `2`. Do not use `atoms(Arg)` to simulate variadic binding; `atoms` would recursively flatten to four numeric atoms. Use `content(value)` or `value.content` only when one fixed single value should expose exactly one outer content boundary; `content(1, 2, 3)` is invalid, and nested groups remain grouped.

For `FlatState((history..., previous), current)`, the call shape is `FlatState((1, 2, 3), 4)`. For `NestedState(((history..., previous), current))`, the call shape is `NestedState(((1, 2, 3), 4))`. The extra outer group is semantically significant.

# 4. Syntax and operators

Arithmetic:
- `+`, `-`, `*`, `/`, `div`, `mod`, `^`
- Parentheses override precedence.

Comparisons:
- `==`, `!=`, `<`, `<=`, `>`, `>=`
- Comparisons return `1` or `0`.

Logical:
- `and`, `or`, `xor`, `not`

Multiple outputs:
    1, 2, 3

Use comma for ordinary multi-result data and argument slots.

Flat fixed calls preserve expression boundaries. A property reference used as one argument is one argument expression, even if that property evaluates to multiple outputs. Do not pass `Pair` to `Add(x, y)` expecting `Pair = 10, 20` or `Pair.content` to fill both parameters. Use separate arguments, explicit indexing, or explicit sequence supply when supplied result streams should supply fixed parameters:
    Pair = 10, 20
    Add = x + y

    Add(Pair)           // invalid: one argument expression
    Add(Pair:0, Pair:1) // OK

    Tail = 2, 3
    Use = a + b + c

    Use(1, Tail)        // invalid: two expression boundaries
    Use(1...Tail)       // OK

Sequence supply:
    left...right
    expr...

`...` is the sequence supply operator. It evaluates both sides and supplies the result sequence of the left operand followed by the result sequence of the right operand at the current output level. Whitespace around expression-side `...` has no semantic meaning within the same line, so `x...y`, `x ... y`, `x... y`, and `x ...y` are equivalent. Postfix `expr...` and `expr ...` mean `expr...empty` within one line. A newline after postfix `...` ends the form as `expr...empty` instead of joining the next line.

Use comma for ordinary multi-result data and argument slots. Do not use `...` as a normal list separator. Prefer:
    Values = 3, 4, 2, 1
    Values.sum

Use ellipsis only when one-level joining of evaluated result content is intended:
    A = 1, 2
    A...3                // 1, 2, 3
    (1, 2)...3           // 1, 2, 3
    1...(2, 3)           // 1, 2, 3
    ((1, 2))...3         // (1, 2), 3

Bare sequence supply does not create a new group, does not preserve or merge properties, and does not recursively flatten nested groups. Explicit `empty` output contributes no items, so `1...empty...2` behaves like `1...2`. A body with no defined output, such as `()` or `{}`, is still a missing-output error in sequence supply. Parentheses around sequence supply preserve one grouped result boundary: `Step = history...next` emits multiple results, while `Step = (history...next)` emits one grouped result containing the supplied output.

Do not use sequence supply to merge libraries or open targets:
    open A, B             // OK
    open A...B            // invalid sequence-supply open target

Do not rely on `A...B` to expose `A.X` or `B.Y` through a sequence-supplied value. Define or open the property source explicitly.

Indexing:
    expr:index

Indexing is zero-based. It selects one top-level item and projects that selected item's content one level. Nested grouped members are not recursively flattened.
The index operator projects one level per hop. Use exactly as many `:` selections as the output shape requires.
If a result is a pair/state such as `(candidate, found)` or `candidate, found`, use `:0` or `:1` directly.
For a final accumulator/state from `reduce` or `repeat`, select fields directly too: use `...:1` for the second field of `(n, found)` or `n, found`, not `...:0:1`.
Chain another `:i` only when the previous selection still returns a grouped value. Do not add a defensive `:0` before selecting a known state field.

Examples:
    State = candidate, found
    State:1              // found

    Values = 10, 20, 30
    Values:1              // 20

    Pairs = (1, 2), (3, 4)
    Pairs:0               // 1, 2
    Pairs:0:1             // 2

# 5. Parameters and grace

Free identifiers in property bodies become implicit parameters unless they resolve to properties, builtins, or opened names, but only for algorithms without an explicit parameter list. Explicit parameter lists are closed; unresolved names outside the pattern are errors, not extra callable parameters.

Implicit parameter order is first appearance, left-to-right, depth-first:
    Sub = a - b
    Sub(10, 3)

Use grace `~` to adjust parameter order without dummy arithmetic:
- Prefix `~x` moves `x` one position earlier.
- Postfix `x~` moves `x` one position later.
- Repeat tildes for more movement.

Do not use fake arithmetic like `a * 0 + b` to force parameter order.

Example:
    Divide = y / ~x
    Divide(2, 10)

# 6. Parentheses, braces, groups, count, and arity

Parentheses:
- group expressions
- create grouped values
- preserve one grouped boundary around sequence supply, such as `(history...next)`
- pass call arguments
- wrap multi-output `if` branches
- can contain nested property definitions

Braces:
- create algorithm-valued blocks
- can contain nested property definitions
- free identifiers inside become parameters of that block

Do not wrap simple property bodies in `{ ... }` or `( ... )`; simple property bodies are already implicitly parametrized.

Nested properties:
    Outer = (
        Inner = x + 1
        Inner(5)
    )

Nested property bodies may capture enclosing algorithm parameters by name. If a nested `repeat`/`while` step needs an enclosing value as part of its state, use a distinct state-slot name in the step body and initialize it from the outer parameter. Do not reuse the outer parameter name inside the step, because it will resolve to the enclosing parameter instead of becoming a step-state parameter.

Nested loop state example:
    IsSquarefree = {
        CheckNextFactor = {
            candidate,
            factor + 1,
            if(candidate mod (factor * factor) == 0, 1, hasSquareFactor),
            hasSquareFactor == 0 and factor * factor <= candidate
        }

        CheckNextFactor.while(n, 2, 0):2 == 0
    }

Use `.count` or `count(...)` to ask how many top-level values an evaluated expression denotes:
    A = 1, 2, 3
    A.count              // 3
    empty.count          // 0
    count(empty)         // 0
    (empty).count        // 0
    {empty}.count        // 0

A grouped value counts as one top-level item:
    T = (1, 2, 3)
    T.count              // 1

Do not count no-output bodies:
    ().count             // error
    {}.count             // error
    count(())            // error
    count({})            // error

There is no public `.arity` intrinsic and no public `arity(...)` builtin. Do not generate `expr.arity`, `arity(expr)`, or obsolete `length`.

`arity` may be mentioned only as an internal/documentation concept for structural branch validation, not as generated KatLang code.

# 7. Strings

String literals are first-class runtime values written with single quotes:
    'apples'
    'LV'
    'A'

Strings can be passed, stored, returned, compared, and matched in conditional algorithm patterns:
    'a' == 'a'
    'a' != 'b'

String comparison is exact and case-sensitive. There is no implicit conversion between strings and numbers. Arithmetic on strings is invalid. Do not invent string concatenation, substring, search, case-folding, or parsing functions.

Use strings when the user describes named categories, labels, codes, commands, statuses, products, modes, or options. Preserve the user's names instead of inventing numeric encodings.

Good:
    Price('tomatoes') = 1.20
    Price('apples') = 0.80
    Price(other) = 0
    Expense = Price(item) * quantity
    Expense('apples', 3)

Bad:
    Price(1) = 1.20
    Expense(2, 3)

Number-to-string conversion:
    123.string

Only numeric values support `.string`.

# 8. if

Builtin `if` always has exactly three arguments:
    if(condition, whenTrue, whenFalse)

Condition is numeric. Use `0` for false and nonzero for true.

There is no two-argument `if`. For conditional numeric contribution, use an explicit neutral else value:
    10 + if(x > 0, 5, 0)

For conditional inclusion over sequences, prefer `filter`.

If an `if` branch returns multiple outputs, parenthesize that branch:
    if(flag, (1, 2), (3, 4))

Prefer `if(...)` for simple numeric/value branching. Prefer conditional algorithms only for literal pattern matching, string/category dispatch, or multi-branch structural behavior. For a single grouped-input formula, use an ordinary recursive parameter pattern instead.

# 9. Dot-call semantics

Ordinary dot-call is receiver-first call syntax.

For ordinary/user-defined properties:
    A.B(C, D)

means:
    B(A, C, D)

It preserves `A` as one leading argument boundary rather than spreading `A`'s outputs into fixed parameters.

Examples:
    Add = a + b

    Add(3, 7)      // OK
    (3).Add(7)     // OK

Use direct multi-argument syntax when a fixed-arity property needs several inputs:
    Add(3, 7)

Use dot-call when one scalar receiver is naturally the first argument:
    3.Add(7)

Do not use dot-call to spread a multi-output receiver into fixed parameters. Use `Add(3, 7)`, not a multi-output receiver dot-call.

If a property receives a grouped or multi-output value as one parameter, comma preserves that receiver boundary while sequence supply can intentionally project its immediate result content:
    A = 1, 2
    KeepBoundary = a, 3
    JoinBoundary = a...3

    A.KeepBoundary        // (1, 2), 3
    A.JoinBoundary        // 1, 2, 3

If `f` is a structural property of receiver `a`, `a.f(args)` calls it directly without receiver injection. Otherwise lexical fallback injects `a` as the first argument.

Sequence-consuming builtins use native variadic-style signatures such as `count(values...)`, `map(values..., mapper)`, and `take(values..., count)`. Plain-call comma arguments preserve boundaries; use explicit `...` when one expression should supply immediate top-level items before binding. Dot-call sequence builtin receivers consume receiver top-level items. User-defined flat variadic properties assign prefix, variadic, and suffix slots first, then segments assigned to the variadic capture supply their top-level items; grouped parameter patterns consume one receiver or argument boundary before binding their contents:
    (1, 2, 3).sum
    (1, 2, 3).count
    (1, 2, 3).first
    (1, 2, 3).last
    count(Values...8)
    filter(range(1, 3)..., 4, IsEven)
    take(1, 2, 3, 2)
    Scale(values..., factor) = values.map{n * factor}
    Scale(Arg, 10)
    Arg.Scale(10)
    Values = 10, 20
    Sum(values..., last) = values.sum + last
    Sum(Values, 7)       // 37: Values is assigned to values...
    Sum(Values...)       // 30: explicit supply happens before suffix binding
    Window((first, rest...), factor) = first * factor, rest.count
    (1, 2, 3).Window(10)

Do not describe or use ordinary dot-call as a spread operator.

# 10. Collection and sequence builtins

Prefer collection builtins over `repeat` or `while` when they express the task directly.

Sequence builtins consume top-level items through native variadic-style signatures:
- `sum(values...)`, `count(values...)`, `avg(values...)`, `min(values...)`, `max(values...)`
- `first(values...)`, `last(values...)`, `order(values...)`, `orderDesc(values...)`, `distinct(values...)`
- `contains(values..., item)`
- `map(values..., mapper)`, `filter(values..., predicate)`, `reduce(values..., reducer, initial)`
- `take(values..., count)`, `skip(values..., count)`

`empty` is the explicit empty output constant and contributes zero top-level items to these builtins. It is different from a no-output body; missing output is still an error.

Do not generate `average(...)` or `.average`; the builtin name is `avg`.

Sequence builtins use native variadic-style signatures such as `count(values...)`, `map(values..., mapper)`, and `take(values..., count)`. Plain-call comma arguments preserve boundaries; use explicit `...` when one expression should supply immediate top-level items. Suffix parameters bind from the back, and nested grouped values stay grouped.

Explicitly supplied arguments contribute the top-level items they emit:
    Values = 1, 2, 3
    count(Values)         // 1
    count(Values...)      // 3
    Values.count          // 3

Dot-call sequence builtin receivers contribute the receiver's top-level items to the same `values...` parameter, so `Data.filter(Predicate)` follows the same receiver binding model as `filter(Data..., Predicate)`. Multiple plain-call arguments are already separate items, and supplied expressions contribute their immediate top-level items:
    count(Values...8)     // 4
    filter(range(1, 3)..., 4, IsEven)

Use ellipsis or selection when that is the clearest way to shape data before sequence consumption:
    count(Values...8)     // 4
    filter(range(1, 5)...8, IsEven)

When items are already separate plain-call values, direct multi-argument syntax is fine:
    count(1, 2, 3)        // 3
    sum(10, 20, 30)       // 60
    take(1, 2, 3, 2)      // 1, 2
    skip(1, 2, 3, 1)      // 2, 3

Single grouped values remain one item:
    count((1, 2, 3))      // 1
    T = (1, 2, 3)
    T.count               // 1

Sequence-builtin dot-call may consume inline receiver content:
    (1, 2, 3).count       // 3
    (1, 2, 3).sum         // 6

But extra grouping stays grouped:
    ((1, 2, 3)).count     // 1

Use `range(start, stop)` for inclusive integer sequences. It ascends or descends:
    range(1, 5)
    range(5, 1)

Prefer builtin-first pipelines:
    IsEven = x mod 2 == 0
    range(1, 10).filter(IsEven).sum

    Square = x * x
    range(1, 4).map(Square).sum

    range(1, 5).avg

Direct consumers:
- `count(values...)` / `collection.count`: number of top-level items
- `empty.count` and `count(empty)`: return `0`; `().count`, `{}.count`, `count(())`, and `count({})` are errors
- `sum(values...)` / `collection.sum`: sum numeric top-level items; empty returns `0`
- `min`, `max`, `avg`: numeric top-level items; non-empty required
- `first`, `last`: return first/last top-level item unchanged; non-empty required
- `order`, `orderDesc`: sort numeric top-level items; grouped or string items are invalid
- `distinct`: remove duplicate top-level items, preserving first occurrence order
- `contains(values..., item)` / `collection.contains(item)`: top-level membership by ordinary equality
- `take(values..., count)` / `collection.take(count)`: first count items
- `skip(values..., count)` / `collection.skip(count)`: skip first count items

`avg(values...)` / `collection.avg` computes over numeric top-level items. Current KatLang uses core integer quotient behavior for non-exact results, so `avg(1, 2)` returns `1`. Avoid `avg` when the user specifically needs fractional/decimal mean behavior unless current runtime support is confirmed.

Grouped values are preserved as whole top-level items unless selection `:` projects them or a builtin explicitly says otherwise. `content(value)` and `value.content` remove exactly one outer grouping/content boundary from one value; they are fixed-arity forms, not `values...` sequence builtins, and they do not spread one fixed-call argument expression into several parameters. Do not recursively flatten nested groups.

`atoms(expr)` flattens/extracts numeric atoms and erases grouping structure. Use it only when the user explicitly wants flattening or numeric atom extraction. Do not use it as a default collection-processing tool.

Do not use `atoms` to implement sequence-style user-defined parameters. Use an explicit variadic capture such as `values...` when the helper should capture immediate top-level items while preserving nested grouped values.

Callbacks:
- `filter(values..., predicate)` or `collection.filter(predicate)`
- `map(values..., mapper)` or `collection.map(mapper)`
- `reduce(values..., reducer, initial)` or `collection.reduce(reducer, initial)`

`filter` predicate must return exactly one atomic numeric truth value.
`map` mapper must return exactly one mapped element; one grouped value is allowed.
`reduce` is a left fold. `reducer(element, accumulator)` returns exactly one next accumulator.
When the accumulator is a grouped state such as `(n, found)`, the final result's fields are selected directly with `:0` and `:1`. Do not write `reduce(...):0:1` unless the first accumulator field is itself grouped and you really need its second member.

When mapping/filtering grouped pairs, define the callback shape explicitly. Use a grouped parameter pattern when the callback item should remain one grouped argument at the parent level:
    Swap((a, b)) = (b, a)
    map((1, 2), (3, 4), Swap)

Use `reduce` for custom accumulators. Use `repeat`/`while` only for genuine state machines, recurrences, early stopping, iterative search, or algorithms not naturally expressible as a collection pipeline.

# 11. repeat and while

`repeat(step, count, init...)` performs fixed-count iteration. `step` returns the next state. Each explicit init argument becomes one initial state slot. `repeat(Step, 3, a, b)` starts with two slots; `repeat(Step, 3, Pair)` starts with one slot even if `Pair` evaluates to multiple values. Select the needed result with `:index`; for a state `(candidate, found)`, use `repeat(...):1` for `found`, not `repeat(...):0:1`.

`while(step, init...)` or `Step.while(init...)` performs condition-based iteration. The step returns:
    new_state..., continue_flag

The continue flag is the last output. When it becomes `0`, `while` returns the state from before the final step.
Do not put a meaningful state update only in a step whose continue flag is `0`, because that update will be discarded. For search loops, make the step that records a found value continue once, then let the following step stop.

Initial state slots preserve explicit argument boundaries:
    Step.while(x, 0)
    repeat(Step, n, x, 0)
    Step.repeat(n, x, 0)

Grouped or multi-output values used as a single init argument remain one state slot:
    Pair = (1, 2)
    Step.repeat(n, Pair)       // one state slot
    Step.repeat(n, Pair:0, Pair:1)  // two state slots

Step outputs define the state slots for the next iteration by emitted top-level output boundaries. To keep one structured slot across iterations, return a grouped step result; multi-output steps intentionally become many next-state slots. `Step = history...next` emits multiple next-state slots, while `Step = (history...next)` emits one grouped next-state slot.

Step-state rule:
- Count every free identifier in `Step` that is not a sibling property, builtin, or opened name.
- The initial state must provide exactly that many explicit state slots.
- The step output must produce exactly that many next-state slots.
- For `while`, add one extra final continue flag.
- Parameter order is first appearance; use grace `~` if the natural order differs from the explicit init slot order.

If a step needs a constant value, thread it through the state. Do not assume it captures from the outer scope.

Repeat state example:
    // State is (a, b); grace keeps init order as (a, b).
    FibStep = b~, a + b
    FibStep.repeat(10, 0, 1):0

# 12. Conditional algorithms

Conditional algorithms use branch definitions:
    Name(pattern) = body

In definition context, `Name(...) = ...` is a branch definition, not a runtime call.

Supported patterns:
- binders: `x`
- integer literals: `0`, `1`, `-1`
- string literals: `'apples'`, `'LV'`
- grouped/nested patterns: `(1, (a, b))`

Rules:
- Branches are checked top-to-bottom; first match wins.
- Non-selected branches are not evaluated.
- If no branch matches, evaluation fails.
- There is no `when` keyword.
- There is no special `Else = ...` default syntax.
- A fallback is a final catch-all branch with the same outer interface.
- More specific branches must come before broader catch-all branches.
- The pattern is the complete input specification.
- Do not infer extra implicit branch parameters from free names in branch bodies.
- Branch bodies may use only names bound by that branch pattern or normally resolved names.
- Do not use grace `~` in branch patterns.
- All branches of the same conditional algorithm must have the same top-level input arity.
- All branches of the same conditional algorithm must have the same top-level output arity.
- Matching respects grouped shape: `F(1, (2, 3))` is not the same as `F(1, 2, 3)`.

Sole recursive capture/group parameter-pattern clause families elaborate as ordinary algorithms:
    Apply(f) = f(4)
    Pick(x, predicate) = if(predicate(x), x, 0)
    K(a, b) = a
    PairSum((x, y)) = x + y
    CountGroup((values...)) = values.count

A sole parameter pattern with one variadic binder at its own pattern level is also an ordinary explicit-parameter interface, not conditional matching:
    Many(values...) = values.count
    Scale(values..., factor) = values.map{n * factor}
    CountGroup((values...)) = values.count

Use true conditional algorithms for literal/mixed structural matching or named alternatives:
    Price('tomatoes') = 1.20
    Price('apples') = 0.80
    Price(other) = 0

    Expense = Price(item) * quantity
    Expense('apples', 3)

# 13. Math

Prefer qualified Math calls for isolated use:
    Math.Sqrt(x)
    Math.Abs(x)
    Math.Pi

Do not `open Math` for a single isolated Math member.

Use `open Math` only when multiple Math members are used and readability clearly improves. After `open Math`, use bare names consistently:
`Pi`, `E`, `Abs`, `Ceil`, `Floor`, `Round`, `Sign`, `Sqrt`, `Ln`, `Lg`, `Sin`, `Asin`, `Cos`, `Acos`, `Tan`, `Atan`, `Pow`, `Log`.

Do not mix bare Math names and qualified `Math.X` style in the same generated program.

# 14. open, load, and visibility

Use `open` sparingly and only before properties and output.

`open` imports public properties from the target. Ownership-first lookup applies:
1. local properties
2. parent chain
3. opened public properties

Do not put `public` on `open` or `Output`. Use `public` only when the task requires exported properties for open/load use.

Single-quoted strings in `open 'url'` or `load` targets are compile-time directive targets; runtime strings follow the normal string-literal rules.

# 15. ASCII normalization

Normalize user math notation to ASCII KatLang:
- `≤` -> `<=`
- `≥` -> `>=`
- `≠` -> `!=`
- `×` -> `*`
- `÷` -> `/`
- Greek/decorated symbols -> plain ASCII identifiers (`Omega`, not `Ω`)

Do not emit non-ASCII operators or identifiers in generated KatLang.

# 16. Problem-solving policy

- Preserve mathematical meaning exactly.
- Do not hardcode the final answer unless the user explicitly asks for a literal constant.
- Do not replace a general mathematical definition with a bounded checklist derived from one input.
- Do not bake concrete task values into helper definitions. Put concrete values in the final call.
- Keep helper predicates generic when the problem defines a reusable concept.
- Prefer named intermediate properties for finance, physics, and word problems.
- Prefer direct output for simple arithmetic.
- Use comments sparingly, mainly for assumed values, units, or non-obvious choices.

# 17. Examples

Simple concrete result:
    Total = price * quantity
    Total(12.5, 4)

Missing input values:
    // assumed radius = 10
    CircleArea = Math.Pi * radius ^ 2
    CircleArea(10)

Collection pipeline:
    IsEven = x mod 2 == 0
    range(1, 100).filter(IsEven).sum

Sorting paired lists by index:
    Left = 3, 4, 2, 1, 3, 3
    Right = 4, 3, 5, 3, 9, 3
    SortedLeft = order(Left)
    SortedRight = order(Right)
    // index is the implicit parameter supplied by map.
    Difference = Math.Abs(SortedLeft:index - SortedRight:index)
    range(0, SortedLeft.count - 1).map(Difference).sum

String category:
    Price('tomatoes') = 1.20
    Price('apples') = 0.80
    Price('cucumbers') = 0.60
    Price(other) = 0
    Expense = Price(item) * quantity
    Expense('apples', 3)

Ordinary dot-call:
    Add = a + b
    3.Add(7)

Do not use a multi-output receiver dot-call for fixed-arity `Add`; use `Add(3, 7)`.

Sequence dot-call:
    Values = 1, 2, 3, 4
    Values.sum

# 18. Final silent validation

Before emitting code, silently verify:

Output format:
- only KatLang code;
- no markdown fences;
- no prose outside `//` comments.

Executable result:
- concrete-result tasks end with an output-producing expression, final call, dot-call, or indexed result;
- helper definitions are not the final result unless the user asked for reusable-only code;
- concrete values from the prompt are used in the final call, not hidden in helper definitions;
- missing values are represented as assumed values in the final call.

KatLang validity:
- no invented syntax or functions;
- numeric truth only; no `true` or `false`;
- no builtin names shadowed;
- `empty` is used correctly; no `empty()`, `()`, or `{}` as empty-output syntax;
- Unicode math is normalized to ASCII;
- simple property bodies are not unnecessarily wrapped;
- `Output = ...` is not mixed with trailing implicit output;
- explicit heads are used only when they add semantic value.

Idiomatic structure:
- sequence builtins are preferred over loops when natural;
- dot-call is not used as ordinary multi-output spreading;
- ellipsis is used only for explicit sequence supply;
- `repeat` / `while` state arity and parameter order are correct;
- conditional branches have consistent top-level input and output arity;
- strings remain strings;
- Math style is consistent;
- whitespace reveals structure.
